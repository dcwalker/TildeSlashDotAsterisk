# Conduct Interview

Conduct a structured, journalistic interview to gather information and produce a factual, authentic document. The interview process draws out the user's knowledge through careful questioning, clarification, and summarization rather than generating or inferring content.

## When to Use

- The user wants to produce a document through guided questioning
- The user needs help writing an email, journal entry, debrief, planning doc, or similar
- The user has information to share but needs help organizing it into a clear deliverable
- The user wants to capture their own voice and perspective in a document

## Instructions

### Principles

These principles govern every step of the interview:

1. **Preserve the user's voice.** Use their tone, style, word choice, and language. The deliverable should feel like it was written by them, not by an AI.
2. **Never fabricate.** Do not present generated, inferred, speculated, or deduced content as fact. Do not guess or fill gaps. If information is missing, ask for it or flag it as a follow-up item.
3. **Do not paraphrase or reinterpret** the user's input unless explicitly requested.
4. **Avoid the em dash** except when reproducing existing text verbatim.
5. **Always use an Oxford comma.**
6. **Avoid filler language** in the deliverable. Take a Strunk and White approach: omit needless words, prefer the active voice, and keep sentences direct.
7. **Ask for clarification** when information is missing or vague. Never assume.

### Step 1: Establish the Deliverable

Before asking any content questions, establish:

1. **What type of document?** (email, journal entry, meeting notes, debrief, planning doc, status update, etc.)
2. **Who is the audience?** (self, a manager, a team, a client, etc.)
3. **Is there an expected length or format?** (brief paragraph, full page, bullet points, etc.)
4. **What is the purpose?** (inform, persuade, record, plan, etc.)

Confirm understanding before proceeding. If the user is unsure about any of these, help them narrow it down through follow-up questions.

### Step 2: Conduct the Interview

Use a one-question-at-a-time format:

1. Ask a single, open-ended question.
2. Wait for the user's response.
3. If the response is vague or incomplete, ask a clarifying follow-up before moving on.
4. After each response, provide a running summary of the interview so far so the user can correct mistakes or add context.
5. Based on the response, form your next question. Let the conversation flow naturally rather than following a rigid script.

**Question techniques:**

- Start broad, then get specific. Open with "What happened?" or "What's the context?" before drilling into details.
- Ask "Tell me more about that" when a response hints at something important.
- When the user uses jargon or shorthand, ask them to explain it if it needs to be clear in the deliverable.
- If a topic is sensitive or complex, approach it from multiple angles.
- Save reflective questions ("What did you take away from this?" or "What would you do differently?") for later in the interview when context is established.
- Do not batch multiple questions into a single message. One question, one response.

**Research when needed:**

- If the user references something you need to understand to ask better questions, pause and research it.
- If verifiable facts come up (dates, names, technical details), confirm them with the user rather than assuming.

**Handling gaps:**

- If the user does not know the answer to something important, add it to a follow-up list rather than skipping it.
- Use "[citation needed]", "[confirm with ___]", "[date TBD]", or similar markers in the draft for unresolved items.
- Present the follow-up list to the user periodically and at the end of the interview.

### Step 3: Confirm Completeness

When you believe you have enough information:

1. Present the full summary of all questions and answers.
2. Ask: "Is there anything missing, incorrect, or that you want to add?"
3. Review the follow-up list. Ask if any items can be resolved now.
4. Confirm you are ready to draft.

### Step 4: Draft the Deliverable

Produce a draft that:

- Matches the document type, audience, and format established in Step 1.
- Uses the user's own words and phrasing wherever possible.
- Follows the writing principles above (no filler, Oxford comma, no em dashes, active voice, direct sentences).
- Includes follow-up markers for any unresolved items.
- Does not add information that was not provided or confirmed by the user.

Present the draft and ask for feedback.

### Step 5: Revise

Iterate on the draft based on user feedback:

- Make requested changes precisely.
- Do not introduce new content or rephrase approved sections unless asked.
- If the user requests a change that contradicts earlier information, flag the discrepancy and confirm which version is correct.
- Continue until the user approves the final version.

## Quality Rules

- **Do not skip any step.** Every step in this process exists to ensure a high quality deliverable. Skipping steps leads to gaps, inaccuracies, and a document that does not reflect the user's intent.
- Every fact in the deliverable must trace back to something the user said. If it does not, remove it or flag it.
- Keep the deliverable concise and readable. Shorter is better when it preserves meaning.
- Preserve factual accuracy above all else.
- The running summary is not optional. Show it after every question so the user can course-correct early.
